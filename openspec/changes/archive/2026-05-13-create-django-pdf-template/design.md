## Context

Currently, the organizational PDF report is generated by reading a static HTML file from `survey/pdf_templates/html-pdf/index.html`. This file is processed as a raw string and passed to WeasyPrint. To support dynamic content, we need to migrate this to a standard Django template and refactor the preview view to render it with context data.

## Goals / Non-Goals

**Goals:**
- Migrate static HTML/CSS report assets to a Django template directory.
- Enable dynamic data injection (company name, participant counts, dates, etc.) via Django's templating engine.
- Refactor the `preview_report_pdf` view to use the new template system.
- Ensure all images and styles load correctly in the generated PDF.

**Non-Goals:**
- Full refactoring of the report's visual design (only the underlying delivery mechanism is changing).
- Migrating the older ReportLab-based PDF generation logic (which is separate from this WeasyPrint report).

## Decisions

- **Template Location**: Move assets to `survey/templates/survey/pdf/`. This follows the project's convention of keeping templates within the relevant app's template directory.
- **Rendering Mechanism**: Use `django.template.loader.render_to_string` to generate the HTML content. This allows us to use all Django template features (tags, filters, context).
- **Asset Resolution**: We will continue to use `base_url` in the WeasyPrint `HTML()` constructor. The `base_url` will point to the physical directory where the template and its assets reside (e.g., `survey/templates/survey/pdf/`) to ensure relative paths for images and CSS work as expected.
- **Dynamic Variable naming**: Standardize context keys such as `company_name`, `total_participants`, `report_date`, etc., to make the template reusable.

## Risks / Trade-offs

- **[Risk] Path resolution for WeasyPrint** → [Mitigation] Carefully set the `base_url` and use relative paths within the template. Avoid using `static` template tags for PDF-only assets if they aren't served by the standard static files app, or ensure they are properly resolved.
- **[Risk] Large template file** → [Mitigation] The existing `index.html` is over 800 lines. We will keep it as a single file for now to maintain parity with the original, but consider splitting it into components using `{% include %}` if further complexity is added.
