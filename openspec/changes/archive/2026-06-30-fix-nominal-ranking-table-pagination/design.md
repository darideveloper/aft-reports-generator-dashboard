## Context

The group report PDF is generated by `utils/group_report_generator.py:58` (`generate_group_report_pdf`), which calls `SurveyCalcsGroupTexts` to compute metrics, builds a context dict, renders `group_report_template.html` via Django's `render_to_string`, and converts to PDF via WeasyPrint's `HTML().write_pdf()`.

The "6. Ranking nominal" section (template lines 328вЂ“367) renders a table of participants ranked by score. The Python side pre-computes `nominal_ranking_chunks` by splitting the participant list into chunks (`NOMINAL_RANKING_CHUNK_SIZE`, currently 16). Each chunk becomes a `<section class="page">` element. CSS enforces `page-break-before: always` on `.page` sections so each chunk starts a new physical page.

**Current problem**: The table has no width constraints (`table-layout: auto` is default), no font-size override (inherits global 11pt), and no padding reduction (inherits global `td { padding: 12px }`). Auto table layout distributes column widths proportionally to content, causing name and position columns to receive unpredictable widths. Long names (32+ characters) or job titles (20+ characters) wrap to 2 lines, making those rows ~48% taller. When multiple rows wrap, the section overflows the Letter page, WeasyPrint spills the last 2вЂ“3 rows to a second page, and the second page has only those overflow rows with massive whitespace.

## Goals / Non-Goals

**Goals:**
- Guarantee that all rows of a nominal ranking chunk fit on a single Letter page, regardless of name or position length (even in the worst case where every row wraps to 2 lines)
- Maintain the existing chunk-based pagination model (one `<section class="page">` per chunk)
- Make `NOMINAL_RANKING_CHUNK_SIZE` env-configurable (default 18) so it can be tuned in prod without code changes
- Preserve all data (no truncation of names or positions)
- Follow the existing CSS pattern in the codebase (`.heatmap-table` already uses `table-layout: fixed`)

**Non-Goals:**
- Changing the chunking strategy to CSS-driven auto-pagination
- Redesigning the nominal ranking table structure (columns, ordering)
- Fixing similar wrapping issues in other tables (heatmap, ranking-table, area-results-table)
- Changing the PDF storage mechanism beyond the bugfix (ContentFile instead of temp file)
- Modifying the Python report generation pipeline
- Changing global font-size or padding defaults

## Decisions

### Decision 1: Use `table-layout: fixed` with explicit `<colgroup>` column widths

**Rationale**: Auto table layout is the root cause вЂ” it gives name and position columns whatever space remains after narrower columns (Ranking, ГЌndice, Nivel, SemГЎforo) claim their minimum width. Fixed layout guarantees every column gets exactly its declared percentage of the table width, every time, regardless of content.

**Alternatives considered**:
- *Reduce chunk size to 10вЂ“12* вЂ” hides the symptom but doesn't fix the root cause. Long names still wrap, just more rooms for the overflow. More pages when printed.
- *`white-space: nowrap` + text truncation* вЂ” prevents wrapping but cuts off names/positions. Unacceptable for a professional report.
- *Truncate names in Python* вЂ” same data-loss concern as CSS truncation.
- *Dynamic chunk size based on max name length* вЂ” over-engineered, harder to test, ignores font-size Г— column-width interaction.

Name column at 35% handles up to ~40 characters вЂ” covers virtually all real names including double-barrel surnames ("MarГ­a Fernanda de la Cruz HernГЎndez" = 38 chars). Extremely long titles ("Subgerente de InnovaciГіn y TransformaciГіn Digital" = 50 chars) may still wrap, but these are rare and the fixed layout ensures the wrapping is predictable.

### Decision 2: Reduce font size to 10pt for this table only

**Rationale**: Smaller font в†’ more characters fit per line в†’ fewer wraps. This is a targeted override via `.nominal-ranking-table { font-size: 10pt; }`, scoped to this table only. The global 11pt standard (`--font-size-normal`) is not changed.

**Trade-off acknowledged**: This deviates from the `standardized-pdf-styling` spec's 11pt requirement for body text. However, the 11pt standard was established before the pagination bug was discovered. The deviation is auditable and contained to one table.

**Alternatives considered**: Keeping 11pt and using wider columns вЂ” would require more than 50% total for name+position, squeezing the other 4 columns to <50% total, making them uncomfortably narrow.

### Decision 3: Reduce cell padding to `8px 6px` (from `12px` all sides)

**Rationale**: At 12px all-sides padding with 11pt font, a single-line row is ~11.3mm tall. At 8px vertical padding with 10pt font, a single-line row is ~8.7mm. A wrapped 2-line row goes from ~15.9mm (old) to ~13.0mm (new). The key insight: with the new fixed layout, FAR FEWER rows will wrap at all. The 6px horizontal padding also helps: less horizontal padding в†’ more space for text content within each column.

|----------|------------------|------------------|

At chunk size 16, the all-2-line worst case does NOT fit the 229mm page. At 18 rows with typical mixed content (mostly 1-line, a few 2-line wraps) it fits comfortably. The env var default of 18 was chosen because it fills the page for single-line rows while being adjustable per-prod. Keeping 12px padding would make rows taller even with 10pt font, reducing capacity.

**Alternatives considered**: Keeping 12px padding в†’ even 15 Г— 2-line would be ~255mm, exceeding the page. The padding reduction is essential.

### Decision 4: Make `NOMINAL_RANKING_CHUNK_SIZE` env-configurable with default 18

**Rationale**: During implementation, the user found that 15 rows left too much whitespace at the bottom of each page. Most rows are single-line (fixed layout + generous columns prevent wrapping), so more rows fit. The fixed overhead (heading + header + table margin в‰€ 30mm) leaves 199mm for data rows. Each single-line row is ~8.7mm, so ~22 rows could fit. Each 2-line row is ~13.0mm. With default 18:
- All single-line: 30 + 18Г—8.7 = 187mm вњ…
- Mixed (3Г—2-line + 15Г—1-line): 30 + 3Г—13.0 + 15Г—8.7 = 200mm вњ…
- All 2-line (worst case, practically impossible with fixed layout): 30 + 18Г—13.0 = 264mm вќЊ

Default 18 fills the page well for realistic data while providing room to adjust per-prod via env var. If prod data shows empty space, operators can raise it without redeploying code.

**Alternatives considered**:
- *Hardcode 18* вЂ” same result but requires code change to adjust
- *Chunk 15* вЂ” safe but leaves too much empty space
- *Algorithmic chunk size* вЂ” over-engineered

### Decision 5: Fix temp file crash вЂ” use `ContentFile` instead of `NamedTemporaryFile`

**Rationale**: The management command `create_group_report.py` wrote PDF bytes to a temp file in `/app/media/temp/`, then read it back to save to Django's storage. In production (Docker), that directory doesn't exist, causing `[Errno 2] No such file or directory`. `ContentFile` wraps bytes in-memory and Django's storage backend writes directly to whatever backend is configured (S3 in prod).

**Alternatives considered**:
- *Create the directory* (`os.makedirs`) вЂ” works but still writes extra file, needs cleanup
- *Use system temp dir* вЂ” works but mixes concerns

## Risks / Trade-offs

- **Font-size deviation**: Using 10pt instead of 11pt for this table deviates from the `standardized-pdf-styling` spec. Mitigation: this is a targeted, documented override. Other tables remain at 11pt. If the spec is later enforced mechanically, this exception should be noted.

- **Extremely long job titles**: Titles exceeding ~37 characters will still wrap to 2 lines in the position column. Mitigation: such titles are rare in practice (<5% of participants). Even when they occur, the fixed layout + reduced sizing means the wrapping is predictable and the chunk still fits one page.

- **Extremely long names**: Names exceeding ~40 characters (very rare; longest typical Spanish double-surname names are 30вЂ“38 chars). Mitigation: same as above вЂ” even if wrapped, the layout remains stable.

- **Dot (semГЎforo) cell height**: The dot is 15px tall, while text cells at 10pt + 8px padding are ~30px tall. The dot is slightly smaller than the cell height. Mitigation: `<span class="dot">` is `display: inline-block` with `vertical-align: middle` (browser default for inline-block), keeping it visually centered. No action needed.

- **Multi-chunk rendering**: The first chunk shows the section title ("6. Ranking nominal"), subsequent chunks do not. This logic is unchanged by the `<colgroup>` addition. Verified: `<colgroup>` is inside `{% for chunk %}` loop, so each chunk-page gets its own `<colgroup>`.
